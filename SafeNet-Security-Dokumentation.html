1. Projektübersicht

SafeNet Security ist eine deutschsprachige (und englische) Cybersicherheits-Lernplattform im Browser. Sie wurde als Schulprojekt entwickelt und ist unter safenet-security.ch öffentlich erreichbar.

Die Plattform verbindet Lerninhalt über Angriffsmethoden mit praktischen Sicherheitstools: Passwort-Analyse, Passwort-Generator, verschlüsselter Tresor, sichere Notizen und einen KI-Assistenten.

1.1 Ziele der Plattform

  • Nutzer über Cyberangriffe informieren (Phishing, Ransomware, Social Engineering, etc.)
  • Praktische Sicherheitstools direkt im Browser anbieten
  • Deutschen und englischen Nutzern eine vollständig lokalisierte Erfahrung bieten
  • Moderne Webtechnologien ohne Frameworks einsetzen (Vanilla JS)

5. Backend – Vercel Edge Functions (api/)

Alle 5 API-Routen laufen als Edge Functions (runtime: 'edge'). Sie starten in Millisekunden weltweit nahe beim Nutzer – kein Cold-Start. CORS ist auf allen Routes auf https://safenet-security.ch beschränkt.

5.1 api/chat.js – KI-Chatbot-Backend

Ablauf einer Chatanfrage:

Nutzer schreibt Nachricht ↓ chat-widget.js ├─ Supabase Session holen → Access Token └─ POST /api/chat { message, history (letzte 10), lang } ↓ api/chat.js (Edge Function) 1. Bearer-Token validieren (Supabase /auth/v1/user) 2. Body parsen: { message, history, lang } 3. Sprache auto-erkennen (DE/EN per Regex) 4. Keyword-Filter (Regex, kein extra API-Aufruf) ↓ SAFE 5. Mistral API: open-mistral-nemo System-Prompt + Few-Shot-Beispiele + History + Nachricht max_tokens: 250, temperature: 0.3, timeout: 10s ↓ 6. JSON { reply: "..." } → Frontend ↓ chat-widget.js zeigt Antwort + speichert History in sessionStorage

5.2 api/vault.js – Passwort-Tresor-Proxy

HTTP-MethodeSupabase-EndpunktFunktion
GET/rest/v1/passwords?select=*&order=created_at.descAlle Passwörter laden
POST/rest/v1/passwordsNeues Passwort speichern
PATCH/rest/v1/passwords?id=eq.{id}Passwort aktualisieren
DELETE/rest/v1/passwords?id=eq.{id}Einzelnes Passwort löschen
DELETE/rest/v1/passwords?user_id=eq.{id}Alle Passwörter löschen

5.3 api/delete-account.js – Account-Löschung

  • Nutzt den SUPABASE_SERVICE_ROLE_KEY (nur im Backend verfügbar, NIE im Frontend)
  • Verifiziert erst den Nutzer via Access-Token
  • Löscht via Supabase Admin-API: DELETE /auth/v1/admin/users/{userId}
  • Supabase löscht durch ON DELETE CASCADE alle Nutzerdaten automatisch

5.4 api/login-benachrichtigung.js – Login-E-Mail

  • Verifiziert den Access-Token bei Supabase
  • Ermittelt IP-Adresse, User-Agent und Zeitstempel (Schweizer Zeit: Europe/Zurich)
  • Sendet E-Mail über Resend API mit Login-Details
  • Verwendet RESEND_API_KEY aus Vercel-Umgebungsvariablen

5.5 api/news.js – RSS-Aggregator

  • Fetcht 5 RSS/Atom-Feeds parallel: The Hacker News, Bleeping Computer, Krebs on Security, Ars Technica, SecurityWeek
  • Parst XML manuell mit Regex (kein Parser-Paket) – unterstützt RSS 2.0 und Atom
  • Normalisiert auf: { title, link, description, date, source }

6. Supabase – Datenbank & Authentifizierung

Supabase-Projekt-URL: https://dygrabyaiyessqmjdprc.supabase.co

6.1 Authentifizierung

  • Email/Passwort Login mit Supabase Auth
  • 2FA/TOTP vollständig integriert (Google Authenticator, etc.)
  • Auth-Level AAL2: Wenn 2FA aktiviert, wird nach dem Login der TOTP-Code verlangt
  • Session in localStorage, persistSession: true, autoRefreshToken: true
  • Tab-Synchronisation via BroadcastChannel API
  • Passwort-Reset-Flow via E-Mail-Link (PASSWORD_RECOVERY-Event)

6.2 Datenbank-Tabellen

TabelleSpaltenZweck
passwordsid, user_id, title, username, password, url, note, created_atPasswort-Tresor
notiz_ordnerid, user_id, name, erstellt_amOrdner-Struktur für Notizen
notiz_titelid, user_id, text, geaendert_amNotiz-Titel (normalisiert)
notiz_inhaltid, user_id, text, geaendert_amNotiz-Inhalt (normalisiert)
notiz_notizenid, user_id, ordner_id, titel_id, inhalt_id, angepinnt, erstellt/geaendert_amHaupt-Notiztabelle mit Fremdschlüsseln

6.3 Row Level Security (RLS)

RLS ist auf ALLEN Tabellen aktiviert. Selbst wenn jemand den Supabase-Anon-Key kennt, kann er nur seine eigenen Daten sehen und bearbeiten.
  • Policy: auth.uid() = user_id für alle Operationen (SELECT, INSERT, UPDATE, DELETE)
  • ON DELETE CASCADE: Wird ein User gelöscht, werden alle seine Daten automatisch mitgelöscht

7. KI-Chatbot – Vollständiger Aufbau

7.1 Frontend: js/chat-widget.js

  • Floating-Action-Button (FAB) unten rechts – nur für eingeloggte User sichtbar
  • Chat-Fenster mit Header (SafeNet-Logo + Titel + Schließen-Button)
  • Nachrichtenliste mit Bot-Avataren und User-Bubbles
  • Eingabefeld (max. 500 Zeichen) + Senden-Button + Enter-Taste
  • Tipp-Indikator ("Denke nach...") während API-Antwort wartet
  • Session-Persistenz: Gesprächsverlauf im sessionStorage
  • Tageslimit: 50 Nachrichten pro User (Rate-Limiting)
  • Hint-Bubble für eingeloggte User nach 2s, für Gäste nach 4s (mit Registrierungs-CTA)

7.2 Sicherheitsfilter

  • Früher: Zweiter Mistral-API-Aufruf zur Klassifizierung → viele falsch-positive Sperren
  • Jetzt: Schneller Keyword-Regex-Check ohne Extra-API-Call
  • Gesperrte Themen: Hitler, Nazis, Holocaust, Terrorismus, Kinderpornografie, Selbstmord-Anleitungen, Drogen kaufen
  • Vorteil: ~50% schnellere Antwortzeit, keine falsch-positiven Sperren

7.3 Mistral-Konfiguration

  • Modell: open-mistral-nemo
  • max_tokens: 250, temperature: 0.3, Timeout: 10 Sekunden
  • System-Prompt: Alle ~30 Seiten-URLs, Verhaltensregeln, Link-Mapping, Sprachregeln
  • Few-Shot-Beispiele: Muster für typische Fragen (Phishing, Rubber Ducky, Passwort-Reset, etc.)
  • Gesprächs-History: Letzte 14 Nachrichten werden mitgesendet

8. Progressive Web App (PWA)

SafeNet Security ist als installierbare Web-App ausgelegt. Nutzer können die Seite zum Homescreen hinzufügen und wie eine native App öffnen.

8.1 manifest.json

  • Name: SafeNet Security, Short Name: SafeNet
  • Display: standalone (kein Browser-Chrome bei Installation)
  • Background Color: #0f172a (Dark-Theme), Theme Color: #3399ff
  • Icons: 48px, 192px, 512px (auch maskable für Android)
  • Shortcuts: Direktlinks zu Generator und Passwort-Analyse aus dem App-Icon

8.2 sw.js – Service Worker

  • Cache-Name: safenet-v4 (wird bei Updates automatisch erneuert)
  • Pre-Cache: Alle kritischen Assets beim ersten Besuch gecacht (HTML, CSS, JS, Partials)
  • Strategie: Network-First – immer frische Daten vom Server, Fallback auf Cache
  • Beim Update: Alle alten Caches werden beim Aktivieren gelöscht (skipWaiting())

9. Sicherheit & Security-Header

9.1 HTTP-Security-Header (vercel.json)

HeaderWert / Zweck
X-Content-Type-Optionsnosniff – verhindert MIME-Type-Sniffing
X-Frame-OptionsDENY – kein Einbetten in iframes (Clickjacking-Schutz)
X-XSS-Protection1; mode=block – Browser-XSS-Filter
Referrer-Policystrict-origin-when-cross-origin
Permissions-Policycamera=(), microphone=(), geolocation=() – alles verboten
Strict-Transport-Securitymax-age=63072000; includeSubDomains; preload – 2 Jahre HTTPS

9.2 Weitere Sicherheitsmaßnahmen

  • Kein innerHTML mit Nutzerdaten (XSS-Schutz) – textContent verwendet
  • Alle API-Routes prüfen Bearer-Token vor jeder Operation
  • Service Role Key niemals im Frontend – nur in Vercel-Umgebungsvariablen
  • Supabase RLS: Jeder User sieht nur seine eigenen Daten
  • CORS eingeschränkt auf safenet-security.ch
  • Have I Been Pwned API: Passwörter per k-Anonymity geprüft (SHA1-Hash, nie Klartext)
  • Eingaben werden vor Supabase-Übergabe validiert (Typ, Länge, Format)

10. Build-Prozess & Deployment

10.1 build.js – Minifizierung mit Terser

  • Alle 67 JS-Dateien in /js/ werden minifiziert und direkt überschrieben
  • Entfernt: alle Kommentare, Whitespace, Zeilenumbrüche
  • Umbenennung (Mangle): Variablen werden zu a, b, c etc.
  • passes: 2 – zwei Optimierungsdurchläufe
  • ES-Modul-Erkennung: Regex auf import{ / import * / import " für korrekte Verarbeitung von Top-Level-Await
  • Ergebnis: 20–30% kleinere Dateien (z.B. tresor.js: 37 KB → 26 KB)
  • Ausführung: node build.js oder npm run minify

10.2 Deployment-Prozess

1. Lokale Entwicklung → node dev-server.js 2. Minifizieren → node build.js 3. Git Push → git add . && git commit && git push 4. Vercel erkennt Push → automatisches Deployment (~30 Sekunden) 5. Vercel führt build.js aus (buildCommand in Vercel-Config) 6. Weltweit auf Vercel Edge-Network aktiv

10.3 vercel.json – Konfiguration

  • cleanUrls: true – URLs ohne .html-Endung funktionieren
  • Rewrites: /api/:path* → Edge Functions
  • Redirect: safe-net-umber.vercel.app → safenet-security.ch (permanent 301)
  • Redirect: /pages/:path* → /de/pages/:path* (Standardsprache Deutsch)
  • Alle Security-Header auf jeder Route gesetzt

13. Zusammenfassung & Fazit

SafeNet Security ist eine vollständig selbst entwickelte Cybersicherheits-Lernplattform. Die wichtigsten technischen Highlights:

  • Vanilla JavaScript: Kein React, kein Vue – alles in reinem JS/HTML/CSS
  • Edge Computing: API-Routen laufen weltweit nahe beim Nutzer (Vercel Edge, kein Cold-Start)
  • Row Level Security: Datenbank-Sicherheit auf Supabase-Ebene, nicht nur im Frontend
  • PWA: Vollständig installierbar als App auf iOS, Android und Desktop
  • KI-Assistent: Integrierter Chatbot mit Kontextwissen über die Plattform (Mistral Nemo)
  • Mehrsprachig: Vollständige DE/EN-Lokalisierung mit ~30 Seiten pro Sprache und 67 JS-Modulen
  • Sicherheits-Header: HSTS, CSP, X-Frame-Options und mehr auf Hosting-Ebene
  • Automatisierte Tests: Playwright E2E-Tests für alle kritischen Seiten

Das Projekt zeigt, dass eine vollwertige Webanwendung mit Authentifizierung, 2FA, Datenbank, KI-Chatbot, E-Mail-Versand, PWA und automatisierten Tests ohne ein einziges Frontend-Framework realisiert werden kann.
